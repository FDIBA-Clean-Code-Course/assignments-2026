1. Intro
	- JS is a single-threaded language
	- whenever we run any JS code a global execution context is created, where code is executed line by line by a thread of execution (with a global memory)
	- this thread of execution goes in and out of functions, which when invoked create their own execution context (with its own local memory)
	- this is easily represented as a stack, which is why we call it the call stack as well
2. Web APIs
	- JS was initially envisioned as a scripting language to enable more interactive web pages, which is why it is missing a lot of functionalities (you cannot start an internal timer, which is used for stuff like debouncing, delaying function executions, etc.)
	- this is why all browsers offer a plethora of functionalities that are available in browser context only - the so-called Web APIs
	- some of the more popular functionalities are the Timer (setTimeout, setInterval) and the XHR, which enables us to fetch data from some other place on the Internet
	- these all accept some form of a callback - below you can see a setTimeout example
	```js
		const doSomethingDelayed = () => {
			console.log('some work');
		}
		
		console.log('first task');
		setTimeout(doSomethingDelayed, 2000);
		console.log('second task');
	```
	- callback (for those that do not remember) is a function passed to another function as an argument to invoke
	- the callback to be executed is passed to the setTimeout function with the delay as a second argument. This is all then passed to the browser functionality Timer, where the timer is started and the callback is returned for execution by the thread of execution in JavaScript after the timer concludes.
	- a question may have popped up by now - how do we ensure that this asynchronous work is ran synchronously? How do we make sure that this delayed callback will not interrupt some synchronous code? Enter the event loop.
3. The event loop 
	- the event loop is a core mechanism of JS which describes how code is queued up for execution. The core loop is thus - given that we have a single thread of execution, at any time we can have only one piece of code running. The event loop constantly checks if the call stack is empty and if it is it will go through all lines of synchronous code before passing the thread to any asynchronous callbacks (such as the Web API callbacks)
	- the Web API callbacks are one such piece of asynchronous work. The callbacks which are returned after the browser has concluded its work are queued up in a specific structured - the macrotask queue or the callback queue. This is indeed a FIFO queue, as opposed to the call stack, which works in a traditional LIFO fashion.
	- only after all synchronous tasks are done, will the event loop poll the macrotask queue for any queued up tasks
	- what about actual asynchronous work as we know it - requests, responses and data from servers that we need to display in tables or graphics on our frontend? Back before ES5, this was done with another Web API - XHR or the XMLHttpRequest API. Below is an example of how this worked, which is far from relevant for how we develop web apps today, but shines a light on what challenges the developers faced in the past
	- this way of handling server requests was imperfect, because it treated the callbacks passed via the event listeners as normal Web API callbacks. This means that they will be potentially queued up after some less important work (such as a timeout callback).
	```js
	const postsUrl = 'https://jsonplaceholder.typicode.com/posts?userId=1';

	const getRequest = new XMLHttpRequest();
	
	getRequest.open('GET', postsUrl);
	
	getRequest.addEventListener("load", (ev) => {
	   console.log("Request successful: ", getRequest.response);
	});
	
	getRequest.addEventListener("error", errorHandler);
	
	getRequest.send();
	```
4. Promises
	- with the advent of ES6 a new way to handle async requests and consume their responses was created as a JS feature - the Promise interface
	- promises return a promise object, which will fill up with some data the moment some asynchronous work has finished and returned either some data or some error
	- we however need a way to consume this data, which is why the Promise object offers us a then method. This method accepts a callback, which will be passed the asynchronous data the moment it is fully present. Then methods are chain-able, so we can queue up more callbacks for more complex cases
	- we can also attach error handlers via the catch method and run some cleanup work in any of the cases with the finally method. All three methods are called consumers.
	 ``` js
	 // This is an example with the browser API fetch function, which returns a Promise object and is used to perfrom API calls
	fetch('resource/url/somewhere')
		.then((responseData) => responseData.json())
		.then((parsedData) => formatData(parsedData))
		.then((formattedData) => consumeDataByService(formattedData))
		.catch((err) => handleError(err))
		.finally(runSomeCleanup);
	 ```
	- we can create our own promise objects with newPromise, which expose a resolve and reject method
	```js
	const myPromise = new Promise((resolve, reject) => {
		// do some work
		resolve(data);
		
		if(someError) {
			reject(new Error('error'));
		}
	});
	
	myPromise.then(consumeData).catch(handleError);
	
	// we can immediately resolve the promise
	const promiseNow = new Promise((resolve, reject) => {
		resolve(2);
	});
	
	myPromise.then((data) => console.log(data)); // logs out 2
	```
	- how does the event loop handle promises? if we assume that we are going to use promises to call our APIs to fetch some highly important user data and we use the callback queue then that means that we can have some less important Web API callback before our important promise data.
	- that's why a new queue is introduced - the microtask queue. The callbacks passed to the consumer methods are queued up in this queue and are handled with a higher priority than the macrotask queue.
	- our updated event loop is thus - the event loop checks for synchronous work if the call stack is empty. When it hits a promise, its callbacks are queued up in the microtask queue, whereas the Web API callbacks are queued up in the macrotask queue. When all synchronous work is done, the microtask queue is polled first until empty and then the loop moves to the macrotask queue.
	- it is theoretically possible to starve the macrotask queue, by queueing up many promise callbacks.
	- callback hell - this condition can arise if we have too many promises that depend on each other. This will lead us to nesting consumer methods within each other, which can quickly become very difficult to debug
	- this condition can be avoided by simply chaining our promises
	``` js
	// Very simple example with only two dependant sequential promises. Now imagine the same but with 3 or 4 more
	const getUsersUrl = "https://jsonplaceholder.typicode.com/users"

	const createUserPostsUrl = (userId) => `https://jsonplaceholder.typicode.com/posts?userId=${userId}`;

	const getUserInformation = (userId, userData) => {
	  const filteredUser = userData.filter((user) => user.id === userId);
  
	  console.log('Filtered user: ', filteredUser);
	 
	  if(filteredUser.length) {
	    return filteredUser[0];
	  }
	  
	  return null;
	}
	
	// callback hell
	fetch(getUsersUrl)
		.then((response) => response.json())
	    .then((userData) => {
	      const userInformation = getUserInformation(1, userData);
	    
	      if(!userInformation) {
	        throw Error ('no user found');
	      }
	    
	      const postsUrl = createUserPostsUrl(userInformation.id);
	      fetch(postsUrl).then(response => response.json())
	        .then((responseData) => console.log(responseData))
	        .catch((err) => console.log(err.message));
	    })
	    .catch((err) => console.error(err.message));
	    
	    
	// avoid the hell by chaining your promises
	fetch(getUsersUrl)
      .then((response) => response.json())
      .then((userData) => getUserInformation(1, userData))
      .then((userInformation) => {
          if(!userInformation) {
            console.log('error case');
            throw Error("no user found");
          }
        
          return createUserPostsUrl(userInformation.id)
      })
      .then((postsUrl) => fetch(postsUrl))
      .then((postsResp) => postsResp.json())
      .then((postsData) => console.log(postsData))
      .catch((err) => console.error(err.message));
	```
	- the Promise constructors offers us two powerful methods to run promises concurrently (start them at the same time i.e.) - Promise.all and Promise.allSettled. They bought take an array of promise objects as an input and return a new promise. The important distinction is that Promise.all will outright fail when a single promise rejects (it will fail with the rejection error of the failed promise), whereas Promise.allSettled will return an array of the promise objects after they all either resolve or reject, allowing us to still consume the promises that resolved successfully. If none of the promises in the Promise.all fail, it will resolve with an array of the promise values.
	```js
	const promise1 = Promise.resolve(1);
	const promise2 = Promise.reject(new Error('Promise rejected'));
	const promise3 = Promise.resolve(3);
	
	Promise.all([promise1, promise2, promise3])
		.then((results) => console.log("All: ", results)) // if all resolved [1,2,3]
		.catch((err) => console.log("All failed with: ", err.message)); // Promise rejected
		
	Promise.allSettled([promise1, promise2, promise3])
		.then((results) => console.log("All settled: ", results));
	// All settled: [ { status: "fulfilled", value: 1}, { status: "rejected", reason: Error object, { status: "fulfilled", value: 3 }]
	```
5. Async / await
	- the ES2017 update introduced a new way of handling promises - with the async / await syntactic sugar
	- the await keyword allows us to "unpack" our promise result; additionally it will "pause" the function execution at this point and will return to it once the promise is available
	- this is only possible inside functions we mark as async
	- this makes our async code more clear, readable and easier to debug
	- here is the above example rewritten with async / await
	```js
	const displayUserPosts = async (userId) => {
	    const usersResponse = await fetch(getUsersUrl);
	    const userData = await usersResponse.json();
	  
	    const userInformation = getUserInformation(userId, userData);
	    
	    if(!userInformation) {
	        throw Error ('no user found');
	     }
	    
	    const postsUrl = createUserPostsUrl(userInformation.id);
	    const postsResponse = await fetch(postsUrl);
	    const postsData = await postsResponse.json();
	    console.log(postsData);
	} 

	try {
	   displayUserPosts(1);
	} catch(err) {
	  console.log(err.message);
	}

	console.log('run sync code 1');
	console.log('run ssync code 2');
	
	// run sync code 1
	// run sync code 2
	// console.log from getUserInformation
	// console.log of posts data // error log
	```
	- in this case pausing the function means that the first time we hit await, the function execution context is exited to handle all the other synchronous code; in essence we are treating everything after the await as a callback passed to then, which is queued up in the microtask queue. This is done for each await we hit as we go through the execution context of the function

	- Promises #2 
	```js 
// Sum multiple parallel promises - Given four promises, which will either resolve with a number or reject with an error, run them parallely and sum them. The sum operation must not fail if any or all promises reject.

// Input
	const promise1 = new Promise((resolve) => setTimeout(resolve(12), 50));
	const promise2 = new Promise((resolve) => setTimeout(resolve(13), 20));
	const promise3 = new Promise((resolve, reject) => setTimeout(reject(new Error("An error was thrown")), 30));
	const promise4 = new Promise((resolve) => setTimeout(resolve(14), 25));
	
// Expected output --> 39

// Solution
	Promise.allSettled([promise1, promise2, promise3, promise4])
	.then((results) => {
	  let sum = 0;
	  results.forEach((promise) => {
	    if(promise.status === 'fulfilled') {
	      sum += promise.value;
	    }
	  })
	 return sum;
	})
	.then((sum) => console.log('Sum of promises: ', sum));
	```
